# Процесс релизов

## Основная модель

- `main` содержит только принятые изменения после review.
- Pull request в `main` проверяются автоматической сборкой.
- После merge в `main` автоматически собирается стабильный релиз.

## Проверка Pull Request

Workflow PR:

- собирает все overlay-проекты
- не публикует APK и не создаёт релиз
- нужен только для проверки, что изменения не ломают сборку

## Preview для переводчиков

Preview собирается только в рамках pull request и не публикуется в GitHub Releases.

Ожидаемый порядок:

1. Сделать fork.
2. Внести изменения в своём fork и открыть pull request в `main`.
3. В комментарии к PR написать `/build`.
4. Дождаться завершения workflow и перейти по ссылке на ZIP-артефакт из ответного комментария в PR.
5. Проверить изменения на автомобиле.
6. Отметить checklist в PR и дождаться review.

## Стабильный релиз

Workflow стабильного релиза:

- запускается на `push` в `main`
- реагирует только на изменения исходников overlays, установочных скриптов и packaging/build-скриптов
- определяет текущий `MAJOR` по git-тегу `major/N`
- автоматически вычисляет следующий `MINOR`
- собирает все overlay APK
- формирует установочный архив
- публикует его в GitHub Releases

## Как менять major версию

Если нужен новый основной ряд версий:

1. Убедитесь, что нужный commit уже находится в `main`.
2. Создайте тег вида `major/1`, `major/2`, `major/3`.
3. Запушьте этот тег в репозиторий.
4. Следующий стабильный релиз автоматически станет `vN.0`.

Пример:

```bash
git checkout main
git pull
git tag major/1
git push origin major/1
```

После этого следующий stable release будет `v1.0`, затем `v1.1`, `v1.2` и так далее.

## Что входит в стабильный релиз

- все overlay APK
- `adb` и сопутствующие файлы
- install/uninstall scripts
- disable-verity scripts
- краткая инструкция по установке и удалению

## Что не входит в релиз

- штатные APK из автомобиля
- preview-пакеты из pull request
- build-артефакты разработчиков
